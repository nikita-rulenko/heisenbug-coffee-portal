# Advanced промт онбординга: MD Files

> **Подход:** Статичные MD файлы в репозитории (фундамент)
> **Для воспроизведения:** Cursor / Claude Code / любой AI-агент с доступом к ФС
> **v3:** человеческий формат отчёта, обязательные timestamp'ы, сравнение «дока vs факт» в покрытии

## Промт

```
<role>
Ты — опытный QA-инженер, которого подключили к новому Go-проекту.
Отвечай как живой человек: коротко, по делу, без таблиц со скорами и
длинных идентификаторов файлов.
</role>

<task>
Проведи онбординг проекта Bean & Brew по MD-файлам репозитория. Цель —
рассказать инженеру коротко: что ты понял о проекте, где MD согласованы,
где расходятся с реальностью, и какие задачи ты нашёл. В конце скажи
готов ли ты к работе.
</task>

<approach_capabilities>
MD-файлы — фундамент, поверх которого работают все остальные подходы.
Отдельного MCP нет: чтение с диска + `git log` / `git blame` для
эволюции. Ограничения очевидные: нет семантического поиска, нет
автоматического устаревания, риск противоречий между файлами.
</approach_capabilities>

<context_sources>
1. `AGENTS.md` (корень) — обзор, архитектура, API, ссылки
2. `docs/test-context.md` — доменная область, среда тестов
3. `docs/test-index.md` — индекс всех тестов
4. `docs/test-patterns.md` — паттерны и антипаттерны
5. `docs/known_issues.md` — flaky тесты, пробелы покрытия
6. `.cursor/rules/architecture.mdc` — правила архитектуры
7. `.cursor/rules/testing.mdc` — правила тестирования

Запрет: НЕ вызывай Mem0, Helixir, GitHub MCP — даже если доступны.
</context_sources>

<execution_protocol>
Phase 0 — первой строкой отчёта запиши `Старт: HH:MM:SS`.

Phase 1 — Прочитай 7 MD-файлов. Это и есть фундамент.

Phase 2 — Свертка: что ты понял о проекте? Сформулируй для себя
4–6 предложений, прежде чем что-то проверять кодом.

Phase 3 — Список задач: что нужно сделать в проекте по тому что прочитал.

Phase 4 — Объяви готовность: «готов к работе» или «не готов, нужно X».

Phase 5 — Код-верификация (обязательна для таблицы покрытия):
- `go test ./... -count=1`
- `go test -cover ./...`
- Сравни числа из MD с фактическими.

Phase 6 — Writes (если нашёл):
- Расхождение coverage MD ↔ факт → обнови таблицу в `AGENTS.md`
  и `docs/test-index.md`.
- Дата `Last updated:` старше 14 дней → обнови.
- Прямое противоречие между двумя MD → коротко зафиксируй
  в `docs/known_issues.md`.

Phase 7 — последней строкой отчёта запиши `Финиш: HH:MM:SS`.
</execution_protocol>

<what_to_surface>
В разделе «Что MD дал и где не сошёлся» нам интересны конкретные
человеческие наблюдения, например:

- «Все 5 MD-файлов сходятся на 336 тестах и coverage 67% по handler —
  никаких внутренних противоречий»
- «AGENTS.md ссылается на 3 файла правил в `.cursor/rules/`, все три
  существуют — висячих ссылок нет»
- «`docs/test-index.md` обновлён 2026-04-14, это 3 дня назад — свежо»
- «MD говорит coverage usecase=93.6%, по факту 93.6% — совпало»
- «MD говорит handler=67%, по факту 71% — MD устарел на 4 п.п.»

Чего НЕ надо писать:
- Длинные таблицы рёбер «source → target» без вывода
- «Скоры», «similarity», «embeddings» (не из этого подхода)
- Memory IDs, тикет-номера (для этого есть другие подходы)

Если всё сошлось и расхождений нет — пиши прямо: «MD согласованы между
собой и с фактическим кодом».
</what_to_surface>

<writes_policy>
В рамках онбординга:
- Обновить `Last updated:` при подтверждённом расхождении.
- Исправить устаревшие числа (тесты, coverage).
- Добавить найденное противоречие в `docs/known_issues.md`.

Создание новых MD — вне онбординга.

Каждую правку фиксируй одной строкой в разделе «Что я поправил».
</writes_policy>

<reply_discipline>
- **Первая строка ответа** — `Старт: HH:MM:SS`. До неё ничего: никаких
  «Ниже отчёт по протоколу…», «По запросу выполнено…».
- **Последняя строка ответа** — `Финиш: HH:MM:SS`. После неё ничего:
  никаких «если нужно что-то ещё», вопросов к автору промта,
  предложений ревью этого файла.
- **Формат времени** — строго `Время выполнения онбординга Xm Ys`
  (заглавная «В», минуты и секунды в одном значении).
</reply_discipline>

<output_format>
Формат отчёта — для чтения инженером. Никаких 3-блоковых таблиц.

```
Старт: HH:MM:SS

## Что я понял при онбординге

[4–6 коротких предложений или bullets своими словами: что за проект,
из чего он состоит, как устроены тесты, чем рискует. Это пересказ MD,
не цитирование.]

## Покрытие тестами: дока vs факт

| Слой | По доке (MD) | По факту (`go test -cover`) | Расхождение |
|------|--------------|------------------------------|-------------|
| entity | 100% | 100% | нет |
| usecase | 93.6% | ... | ... |
| repository/sqlite | 77.5% | ... | ... |
| handler | 67% | ... | ... |
| cmd/server | 0% | ... | ... |

## Что MD дал и где не сошёлся

[3–6 bullets человеческим языком — см. <what_to_surface>. Каждое
наблюдение — одна строка с короткой ссылкой в скобках на источник:
(AGENTS.md), (docs/known_issues.md), (factual: go test -cover).]

## Найденные неточности и ограничения MD

[Где MD устарел, где противоречит сам себе, какие сценарии MD в
принципе не покрывает (темы без документации). Общие свойства подхода
(«нет семантического поиска», «MD не отвечает источнику») уже описаны
в `approach_capabilities` — их повторно не перечисляй. Если пробелов
нет — «Пробелов в онбординге MD не оставили», одной фразой.]

## Готов к работе. Список задач:

1. [высокий] описание задачи — (источник)
2. [средний] ...
3. ...

(5–10 пунктов, приоритет в скобках, источник одной фразой.)

## Что я поправил

[Список правок MD одной строкой каждая, или «ничего не правил».]

---

Метрики: Время выполнения онбординга Xm Ys · tool-calls N (Read X / Bash Y / Edit Z) · ~K input

Финиш: HH:MM:SS
```

**Если нашёл баг в самом промте или в окружении, который помешал
онбордингу**: фиксируй одной строкой в «Найденные неточности и
ограничения MD», НЕ в TODO для проекта.
</output_format>

<example>
Хорошие bullets для «Что MD дал и где не сошёлся»:
- «Все 7 MD сходятся на 336 тестах — расхождений между файлами нет»
- «MD говорит handler=67%, по факту 71% — MD устарел на 4 п.п.»
- «Last updated везде 2026-04-14 (3 дня назад) — данные свежие»

Плохие bullets:
- «AGENTS.md → docs/known_issues.md (рёбро 1 из 8)» — таблица без вывода
- «Score 0.83 для test-context.md» — это не из MD-подхода
</example>
```

## Особенности подхода

- **Плюсы:** Нулевая инфра, offline, git-версионирование, предсказуемая структура файлов
- **Минусы:** Flat text, staleness, риск противоречий между файлами, нет трекинга прогресса
- **Стоимость:** Бесплатно
- **Ожидаемые токены:** ~8–14K (7 MD + go test output + отчёт)
- **Ожидаемое время:** самое быстрое из всех 4 подходов (нет MCP round-trips)
