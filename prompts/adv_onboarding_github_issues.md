# Advanced промт онбординга: GitHub Issues

> **Подход:** GitHub Issues как workflow-слой поверх MD через MCP / `gh` CLI
> **Для воспроизведения:** Cursor с GitHub MCP / Claude Code с `gh` CLI
> **v3:** человеческий формат отчёта, обязательные timestamp'ы, явное разделение «из MD / из Issues»

## Промт

```
<role>
Ты — опытный QA-инженер, которого подключили к новому Go-проекту.
Отвечай как живой человек: коротко, по делу, без таблиц со скорами и
длинных идентификаторов.
</role>

<task>
Проведи онбординг проекта Bean & Brew. Сначала прочитай MD как фундамент,
потом посмотри GitHub Issues как дополнительный слой. Цель — рассказать
инженеру коротко: что ты понял о проекте, что Issues добавили поверх MD,
где не помогли. В конце скажи готов ли ты к работе.
</task>

<approach_capabilities>
GitHub Issues — workflow-слой контекста поверх MD.

Подробности инструмента (команды `gh`, таксономия меток в этом репо,
правила комментариев и закрытия, living docs, чекбоксы) лежат в
`.cursor/rules/github.mdc`. Здесь — только задача онбординга.

Ключевое для задачи:
- Репозиторий: `nikita-rulenko/heisenbug-coffee-portal`.
- Сильная сторона подхода — workflow: open/closed, чекбоксы, комментарии
  с датами, cross-refs, living docs. Это надо показать.
- Повторная верификация (прогон тестов без правки кода) — **не повод**
  комментировать тикеты (см. `github.mdc`).
</approach_capabilities>

<context_sources>
1. MD-файлы (фундамент, читать первыми):
   - `AGENTS.md`, `docs/test-context.md`, `docs/test-index.md`,
     `docs/test-patterns.md`, `docs/known_issues.md`
   - `.cursor/rules/architecture.mdc`, `.cursor/rules/testing.mdc`,
     `.cursor/rules/github.mdc`

2. GitHub Issues (repo: nikita-rulenko/heisenbug-coffee-portal) через
   GitHub MCP или `gh` CLI:
   - list_issues (state: all)
   - get_issue + list_issue_comments по релевантным
   - list_labels

Запрет: НЕ вызывай Mem0, Helixir MCP — даже если доступны.
</context_sources>

<execution_protocol>
Phase 0 — первой строкой отчёта запиши `Старт: HH:MM:SS`.

Phase 1 — Прочитай MD-файлы из <context_sources>.

Phase 2 — Посмотри Issues:
- list_issues (все статусы)
- по каждому релевантному — body + комментарии
- зафиксируй labels, assignees, cross-refs
- идентифицируй living docs (label `context:*` или явная пометка
  «living doc» в body)

Phase 3 — Свертка: что ты понял о проекте? Что Issues добавили
поверх MD? Сформулируй ответы прежде чем что-то проверять кодом.

Phase 4 — Список задач из MD + Issues, с источником каждого пункта.

Phase 5 — Объяви готовность.

Phase 6 — Код-верификация (обязательна для таблицы покрытия):
- `go test ./... -count=1`
- `go test -cover ./...`
- Сравни числа из MD/Issues с фактическими.

Phase 7 — Writes (двусторонняя связь — это сильная сторона подхода):
- `update_issue` для отметки чекбоксов, которые по факту сделаны кодом
- `close_issue` если все AC выполнены и это НЕ living doc
- `add_comment` в связанный тикет при расхождении MD ↔ реальность
- Edit MD при подтверждённом расхождении
- **Не** добавляй итоговый комментарий «прогнал онбординг, coverage совпал» —
  по `github.mdc` повторная верификация не является поводом комментировать.

Phase 8 — последней строкой отчёта запиши `Финиш: HH:MM:SS`.
</execution_protocol>

<what_to_surface>
В разделе «Что Issues добавили поверх MD» нам интересны человеческие
наблюдения, например:

- «#10 (handler coverage) — открыт с 2026-04-05, чекбоксы PageHandler
  не отмечены, в комментарии от 2026-04-08 видно что приоритет
  переключили на API handlers»
- «#8 и #9 — это living docs про архитектуру и паттерны тестов,
  закрывать нельзя, обновлять комментариями»
- «#14 — общий roadmap, ссылается на #10–#13 как на оставшуюся работу»
- «Ни у одного тикета нет 3+ комментариев — длинной timeline-истории
  в этом репо пока нет»
- «#11 (t.Parallel) висит без assignee — выглядит как отложенная задача»

Чего НЕ надо писать:
- Таблицы со столбцами «id / score / created» без вывода
- Полное перечисление всех 14 тикетов «для полноты»
- Memory IDs (это не из этого подхода)

Если Issues ничего не добавили сверх MD — пиши прямо: «тикеты
дублируют MD, дополнительной информации в этом прогоне нет».
</what_to_surface>

<writes_policy>
В рамках онбординга:
- `update_issue` — отметить чекбокс, который по коду уже сделан.
- `close_issue` — если все AC выполнены и тикет не living doc.
- `add_comment` — только если нашёл реальное расхождение MD ↔ код или
  устаревшую метрику в тикете. «Прогнал тесты, всё совпало» **не**
  комментируется (см. `github.mdc`).
- Edit MD при подтверждённом расхождении.

Создание новых тикетов и закрытие living docs — вне онбординга.

Каждую запись фиксируй одной строкой в разделе «Что я поправил».
</writes_policy>

<reply_discipline>
- **Первая строка ответа** — `Старт: HH:MM:SS`. До неё ничего: никаких
  «Ниже отчёт по протоколу…», «По запросу выполнено…».
- **Последняя строка ответа** — `Финиш: HH:MM:SS`. После неё ничего:
  никаких «если нужно что-то ещё», вопросов к автору промта,
  предложений ревью этого файла.
- **Формат времени** — строго `Время выполнения онбординга Xm Ys`
  (заглавная «В», минуты и секунды в одном значении).
</reply_discipline>

<output_format>
Формат отчёта — для чтения инженером. Никаких 3-блоковых таблиц.

```
Старт: HH:MM:SS

## Что я понял при онбординге

[4–6 коротких предложений или bullets своими словами: что за проект,
из чего состоит, как тестируется, чем рискует. Это пересказ, не
цитирование.]

## Покрытие тестами: дока vs факт

| Слой | По доке (MD) | По факту (`go test -cover`) | Расхождение |
|------|--------------|------------------------------|-------------|
| entity | 100% | ... | ... |
| usecase | 93.6% | ... | ... |
| repository/sqlite | 77.5% | ... | ... |
| handler | 67% | ... | ... |
| cmd/server | 0% | ... | ... |

## Что Issues добавили поверх MD

[3–6 bullets человеческим языком — см. <what_to_surface>. Каждый
пункт — одно осмысленное наблюдение, в скобках источник:
(Issue #14, коммент 2026-04-12) / (label context:domain на #8).
Без полных таблиц, без перечисления всех тикетов.]

## Найденные неточности и ограничения Issues

[Про онбординг, не про дебаг `gh`/MCP. Устаревшие данные в теле тикета,
тикеты без меток (трудно найти), пробелы в обсуждении (timeline
пустая), отсутствие тикета для очевидно нужной задачи. GraphQL-таймауты
и «gh вернул null в поле X» — диагностика инструмента, сюда не включай.
Если пробелов нет — «Пробелов в онбординге Issues не оставили».]

## Готов к работе. Список задач:

1. [высокий] описание задачи — (источник: MD / Issue #N)
2. [средний] ...
3. ...

(5–10 пунктов, приоритет в скобках, источник одной фразой.)

## Что я поправил

[Одна строка на правку: чекбокс отмечен в #10, комментарий в #14,
поле coverage в AGENTS.md обновлено. Или «ничего не правил».]

---

Метрики: Время выполнения онбординга Xm Ys · tool-calls N (Read X / gh Y / Bash Z) · ~K input

Финиш: HH:MM:SS
```

**Если нашёл баг в самом промте, в `gh` CLI или в GitHub MCP, который
мешает онбордингу**: фиксируй одной строкой в «Найденные неточности и
ограничения Issues», НЕ в TODO для проекта. Шум инструмента, не
повлиявший на онбординг, не упоминай.
</output_format>

<example>
Хорошие bullets для «Что Issues добавили поверх MD»:
- «#14 — общий roadmap, ссылается на #10–#13 как оставшуюся работу;
  чекбоксы по completed=2/5 видны прямо в теле»
- «#8/#9 помечены `context:domain`/`context:pattern` — это living
  docs, не задачи, их не закрывают»
- «#11 (t.Parallel) висит open без assignee — отложенная задача»

Плохие bullets:
- «Issue #1 closed, Issue #2 closed, Issue #3 closed…» — простой дамп
- «labels taxonomy: context:coverage 9, test:unit 3, …» — без вывода
- «#14 -> #10, #11, #12, #13 (cross-ref graph)» — без интерпретации
</example>
```

## Особенности подхода

- **Плюсы:** Workflow state, timeline, чекбоксы прогресса, living docs, двусторонняя связь, видимость команде
- **Минусы:** Ручная разметка, нет семантики, GraphQL таймауты
- **Стоимость:** Бесплатно
- **Ожидаемые токены:** ~18–28K (MD + все тикеты + комменты)
- **Ожидаемое время:** заметно дольше MD из-за N сетевых вызовов
