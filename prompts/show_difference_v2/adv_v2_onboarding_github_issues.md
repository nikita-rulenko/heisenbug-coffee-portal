# Advanced v2 промт онбординга: GitHub Issues

> **Подход:** GitHub Issues как workflow-слой поверх MD через MCP / `gh` CLI
> **Для воспроизведения:** Cursor с GitHub MCP / Claude Code с `gh` CLI
> **v2 отличие от v1:** verification-first методология + discrepancy banner
> в самом верху отчёта. Утверждения из тел тикетов (AC, проценты, счётчики,
> ссылки на файлы) обязаны быть сверены с реальным состоянием кода.
> Расхождение выделяется крупно, не прячется в прозу.

## Промт

```
<role>

Отвечай как живой человек: коротко, по делу, без длинных идентификаторов.
Дисциплина: ни одна цифра/имя из MD или тикетов не попадает в отчёт
без сверки с реальностью.
</role>

<task>
Проведи онбординг проекта Bean & Brew. Читай MD как фундамент,
Issues — как workflow-слой поверх. Параллельно обязательно сверь
каждое проверяемое утверждение (из MD и из тел/комментов тикетов)
с реальным состоянием кода. Любое расхождение выдели крупно в самом
начале отчёта — инженер должен увидеть это в первый же экран.
</task>

<approach_capabilities>
GitHub Issues — workflow-слой контекста поверх MD.

Подробности инструмента (команды `gh`, таксономия меток в этом репо,
правила комментариев и закрытия, living docs, чекбоксы) лежат в
`.cursor/rules/github.mdc`. Здесь — только задача онбординга.

Ключевое для задачи:
- Репозиторий: `nikita-rulenko/heisenbug-coffee-portal`.
- Сильная сторона подхода — workflow: open/closed, чекбоксы, комментарии
  с датами, cross-refs, living docs.
- Особое внимание: тикет МОГ быть закрыт / чекбоксы отмечены, пока
  реальность разошлась с телом. Доверяй коду, не телу тикета.
</approach_capabilities>

<context_sources>
1. MD (фундамент):
   - `AGENTS.md`, `docs/test-context.md`, `docs/test-index.md`,
     `docs/test-patterns.md`, `docs/known_issues.md`
   - `.cursor/rules/architecture.mdc`, `.cursor/rules/testing.mdc`,
     `.cursor/rules/github.mdc`

2. GitHub Issues (repo: nikita-rulenko/heisenbug-coffee-portal) через

3. Реальность (ground truth):

Запрет: НЕ вызывай Mem0, Helixir MCP — даже если доступны.

<methodology name="Verification-first onboarding">
Железный протокол. Не отклоняйся, не сокращай фазы.

**Phase 0** — первой строкой запиши `Старт: HH:MM:SS`.

**Phase 1 — Extract.** Прочитай MD и Issues из <context_sources>.
Выписывай ПРОВЕРЯЕМЫЕ утверждения (для себя, не в отчёт).
Проверяемое = конкретное и сравнимое: число, процент, счётчик,
имя файла, имя функции, acceptance criteria типа «≥85% coverage»,
«добавлены тесты X Y Z», «слой A содержит N endpoints».

Минимум **8** утверждений суммарно. Каждое тегируй источником:
`[MD]` или `[Issue #N]` / `[Comment в #N]`.

Если извлечено меньше 8 — зафиксируй «источник слишком абстрактный,
извлечено N утверждений» в banner и продолжай.

**Phase 2 — Reality probes.** Независимо от того, что говорят MD и
Issues, выполни базовый набор «прощупываний» рабочего дерева.
Минимум 4 из:

- `go test -cover ./...`
- `grep -rEn "^func Test" internal/ cmd/ | wc -l`
- `grep -rEn "t\.Run\(" internal/ cmd/ | wc -l`
- `ls internal/`
- `grep -rEn "r\.(Get|Post|Put|Delete|Handle)" internal/handler cmd/`
- `Read` seed-файла и ручной подсчёт, если seed упомянут где-то

Цель — поймать то, о чём ни MD, ни Issues не говорят (новые
пакеты/тесты/роуты, удалённые модули).

**Phase 3 — Verify.** Для КАЖДОГО утверждения из Phase 1 выполни
конкретную проверку (команда / Read). Заполни trace (в отчёт):

| # | Утверждение | Источник | Проверка | Результат | Verdict |

Verdict: `✅ MATCH` / `⚠ MISMATCH` / `◌ NOT FOUND`.

Особое внимание: если тикет помечен `closed` и его AC содержит
проверяемую цифру — обязательно проверь, соответствует ли факт AC.
Закрытый ≠ сделанный.

**Phase 3.5 — Rollback-детекция.** Для каждой MISMATCH-строки
проверь: «это `doc_drift` (тикет отстал от кода) или предыдущий
прогон уже чинил это, а потом откатили (`rolled_back_fix`)?» Сигналы
у GH Issues — специфичные:
- Тикет `closed`, но AC по коду не выполнен → классический
  «пролечил и откатил» (closed ≠ done). Уже разбиралось, но в Cause
  это именно `rolled_back_fix`.
- В комментариях тикета есть «fixed in commit abc», а `git log`
  показывает последующий revert.
- В связанном коммите есть слово `revert` / `rollback`.
- `git log --oneline -20 -- <файл>` показывает пару commit → revert.

Пометь в «Результат» в скобках: `(rolled_back_fix)` /
`(doc_drift)` / `(unknown)`. Полный причинный разбор — прерогатива
Helixir-подхода.

**Phase 4 — Cross-check probes.** Если Phase 2 показала что-то,
о чём ни MD, ни Issues не говорят, добавь строку в trace с
Verdict `⚠ MISMATCH` категории `source_behind_reality`.

**Phase 5 — Banner.** Из trace посчитай N_total / N_match /
N_mismatch / N_not_found. Сформируй banner (формат в <output_format>).

**Phase 6 — Отчёт.** Пиши разделы в порядке из <output_format>.
Любая цифра в тексте — со ссылкой `(trace #N)`.

**Phase 7 — Self-audit.** Перечитай черновик. Каждая цифра — с
номером trace. Banner-числа = фактическое число MATCH/MISMATCH/NOT_FOUND.

**Phase 8** — последней строкой `Финиш: HH:MM:SS`.
</methodology>

<writes_policy>
В v2 онбординг = **репорт, не фикс**.

ЗАПРЕЩЕНО во время онбординга:
- Редактировать MD/доки (даже если устарели).
- `update_issue` для отметки чекбоксов (даже если по коду сделано).
- `close_issue`.
- `add_comment` с пересказом «прогнал тесты, вот что».

Все найденные правки — списком в разделе «Что требует правки»,
без применения. Причина: изменение тикетов/MD во время сессии ломает
изоляцию между подходами (следующий подход увидит уже прибранную
картину и пропустит расхождение).

Исключение: если нашёл РЕАЛЬНУЮ ошибку в самом промте или в `gh`
CLI / MCP, которая помешала выполнить онбординг — фиксируй в
«Найденные неточности и ограничения Issues», одной строкой.
</writes_policy>

<reply_discipline>
- **Первая строка** — `Старт: HH:MM:SS`. До неё ничего.
- **Следующий блок** — banner (см. <output_format>). ПЕРЕД «Что я понял».
- **Последняя строка** — `Финиш: HH:MM:SS` или Appendix.
- **Формат времени** — `Время выполнения онбординга Xm Ys`.
</reply_discipline>

## Особенности подхода

- **Плюсы:** Workflow state, timeline, чекбоксы прогресса, living docs, двусторонняя связь, видимость команде
- **Минусы:** Ручная разметка, нет семантики, GraphQL таймауты, «closed» не гарантирует «сделанное»
- **Стоимость:** Бесплатно
- **Ожидаемые токены:** ~20–30K (MD + тикеты + комменты + probes + trace)
- **Ожидаемое время:** заметно дольше MD из-за N сетевых вызовов, +15–20% на verify-проход относительно v1
