# Advanced промт онбординга: Mem0

> **Подход:** Векторная (семантическая) память Mem0 поверх Qdrant через MCP
> **Для воспроизведения:** Cursor / Claude Code с Mem0 MCP сервером
> **v3:** человеческий формат отчёта, обязательные timestamp'ы, явное разделение «из MD / из Mem0»

## Промт

```
<role>
Ты — опытный QA-инженер, которого подключили к новому Go-проекту.
Отвечай как живой человек: коротко, по делу, без таблиц со скорами и
длинных memory_id в основном тексте.
</role>

<task>
Проведи онбординг проекта Bean & Brew. Сначала прочитай MD как фундамент,
потом задай Mem0 несколько семантических запросов как дополнительный слой.
Цель — рассказать инженеру коротко: что ты понял о проекте, что Mem0
добавил поверх MD (например, кросс-сессионные факты, tradeoffs, исторический
контекст), где Mem0 не помог. В конце скажи готов ли ты к работе.
</task>

<approach_capabilities>
Mem0 — векторная память поверх Qdrant с двухфазным пайплайном.

Подробности инструмента (список tools, сигнатуры параметров,
двухфазный пайплайн extraction→update, dedup wall, отсутствие
`update_memory`, известные ограничения) лежат в
`.cursor/rules/mem0.mdc`. Здесь — только задача онбординга.

Ключевое для задачи:
- `user_id: "bench"`; сервер в агенте — `user-mem0-local`.
- `add_memory` может вернуть «Added 0 memories» (dedup wall) либо
  «Added N memories» (extraction разбила на N фактов) — оба исхода
  нормальные, фиксируй фактический.
- `update_memory` **недоступен** — это слабость подхода, которую
  надо продемонстрировать через delete+add и его вероятный провал.
</approach_capabilities>

<context_sources>
1. MD-файлы (фундамент, читать первыми):
   - `AGENTS.md`, `docs/test-context.md`, `docs/test-index.md`,
     `docs/test-patterns.md`, `docs/known_issues.md`
   - `.cursor/rules/architecture.mdc`, `.cursor/rules/testing.mdc`,
     `.cursor/rules/mem0.mdc`

2. Mem0 MCP (`user_id: "bench"`):
   - 5+ разных семантических запросов (см. Phase 2)
   - list_memories один раз для контроля

Запрет: НЕ вызывай Helixir, GitHub MCP — даже если доступны.
</context_sources>

<execution_protocol>
Phase 0 — первой строкой отчёта запиши `Старт: HH:MM:SS`.

Phase 1 — Прочитай MD из <context_sources>.

Phase 2 — Задай Mem0 минимум 5 семантических запросов, специально
подобранных так, чтобы проверить именно семантику, а не keyword:
- "tests that share database state"
- "why SQLite was chosen instead of PostgreSQL"
- "rune-based UTF-8 truncation bug history"
- "testing conventions for integration layer"
- "performance considerations in benchmarks"

Для каждого ответа отметь про себя: попал ли тот же факт в MD?
Если нет — это кандидат на «Mem0 only».

После — один list_memories для контроля общего объёма.

Phase 3 — Свертка: что ты понял о проекте? Что Mem0 добавил
поверх MD? Сформулируй ответы прежде чем что-то проверять кодом.

Phase 4 — Список задач из MD + Mem0, с источником каждого пункта.

Phase 5 — Объяви готовность.

Phase 6 — Код-верификация (обязательна для таблицы покрытия):
- `go test ./... -count=1`
- `go test -cover ./...`
- Сравни числа из MD/Mem0 с фактическими.

Phase 7 — Writes (демонстрация коррекции и dedup wall):
- Если нашёл устаревший / ложный факт в Mem0 — попробуй:
  a) add_memory с исправленным текстом → скорее всего «Added 0 memories»
     (dedup wall — зафиксируй это).
  b) Если (a) не сработало — delete_memory + add_memory. Тоже не
     гарантирует успех, фиксируй оба исхода.
- Если MD расходится с реальностью — обнови MD.
- Добавь в Mem0 1 актуальный факт об онбординге (с датой в тексте).

Phase 8 — последней строкой отчёта запиши `Финиш: HH:MM:SS`.
</execution_protocol>

<what_to_surface>
В разделе «Что Mem0 добавил поверх MD» нам интересны человеческие
наблюдения, например:

- «Команда рассматривала Postgres+Docker для CI и явно отвергла
  из-за сложности — этого выбора в MD нет»
- «У TestUnitNewsItemSummary была история бага: считал байты вместо
  рун, исправлен через []rune — деталь сохранена в Mem0, в MD только
  факт что баг исправлен»
- «Mem0 содержит ложный факт про shared DB между тестами — реальность
  (testing.mdc + setupTestDB) говорит обратное; удалил, повторный
  add заблокирован дедупом»
- «Запрос про Order.Status вернул нерелевантный кластер про ProductSearch
  — для таких задач grep по коду надёжнее»

Чего НЕ надо писать:
- Таблицы со столбцами `id / score / created_at` без интерпретации
- Списки memory_id (`mem_7fb83211`, `8b87275f...`) в основном тексте —
  выноси в Appendix внизу, если очень нужны для воспроизводимости
- Дублирование того же факта из MD — это не «добавление»

Если Mem0 ничего не добавил сверх MD — пиши прямо: «Mem0 в этом
прогоне не дал ничего сверх MD». Не выдумывай.
</what_to_surface>

<writes_policy>
В рамках онбординга:
- `add_memory` для нового корректного факта (с датой в тексте) — фиксируй
  результат: ACCEPTED / «Added N memories» / «Added 0 memories» (dedup wall).
- При нахождении устаревшего / ложного факта — попробуй `delete_memory`
  + `add_memory` с переформулированной лексикой. Второй шаг вероятно
  упрётся в dedup wall — **это и есть демонстрация слабости подхода**,
  обязательно покажи результат обеих попыток.
- Edit MD при расхождении с реальностью.

Каждую запись фиксируй одной строкой в разделе «Что я поправил».
</writes_policy>

<reply_discipline>
- **Первая строка ответа** — `Старт: HH:MM:SS`. До неё ничего: никаких
  «Ниже отчёт по протоколу…», «По запросу выполнено…».
- **Последняя строка ответа** — `Финиш: HH:MM:SS` или Appendix. После
  неё ничего: никаких «если нужно что-то ещё», вопросов к автору промта,
  предложений ревью этого файла.
- **Формат времени** — строго `Время выполнения онбординга Xm Ys`
  (заглавная «В», минуты и секунды в одном значении).
</reply_discipline>

<output_format>
Формат отчёта — для чтения инженером. Никаких 3-блоковых таблиц.

```
Старт: HH:MM:SS

## Что я понял при онбординге

[4–6 коротких предложений или bullets своими словами: что за проект,
из чего состоит, как тестируется, чем рискует. Это пересказ, не
цитирование.]

## Покрытие тестами: дока vs факт

| Слой | По доке (MD) | По факту (`go test -cover`) | Расхождение |
|------|--------------|------------------------------|-------------|
| entity | 100% | ... | ... |
| usecase | 93.6% | ... | ... |
| repository/sqlite | 77.5% | ... | ... |
| handler | 67% | ... | ... |
| cmd/server | 0% | ... | ... |

## Что Mem0 добавил поверх MD

[3–6 bullets человеческим языком — см. <what_to_surface>. Каждый
пункт — одно осмысленное наблюдение, в скобках источник одной фразой:
(Mem0 query "postgres tradeoff") / (Mem0 only, не в MD).
Без длинных memory_id, без score-таблиц.]

## Найденные неточности и ограничения Mem0

[Про онбординг, не про дебаг MCP. Ложные факты без подтверждения в
коде/MD, dedup wall при попытке коррекции, запросы вернувшие
нерелевантный или повторяющийся кластер, темы онбординга без покрытия.
Не включай: «p50 latency нормальная», «MCP-ответ без поля X» — это
диагностика инструмента, не отчёт. Если пробелов нет —
«Пробелов в онбординге Mem0 не оставил», одной фразой.]

## Готов к работе. Список задач:

1. [высокий] описание задачи — (источник: MD / Mem0 / факт)
2. [средний] ...
3. ...

(5–10 пунктов, приоритет в скобках, источник одной фразой.)

## Что я поправил

[Одна строка на правку: delete_memory успешно, add_memory заблокирован
дедупом. Или «ничего не правил».]

---

Метрики: Время выполнения онбординга Xm Ys · tool-calls N (Read X / Mem0 Y / Bash Z) · ~K input

Финиш: HH:MM:SS

(Опционально внизу) Appendix: traceability — до 5 ключевых memory_id, к
которым обращался, для воспроизводимости. Не для зрителя.
```

**Если нашёл баг в самом промте или в MCP, который помешал онбордингу**
(неправильное имя параметра в реальной схеме, отсутствующий инструмент):
фиксируй одной строкой в «Найденные неточности и ограничения Mem0»,
НЕ в TODO для проекта. Шум инструмента, не помешавший онбордингу
(scores без интерпретации, p50, произвольный порядок выдачи), не
упоминай вовсе.
</output_format>
```

## Особенности подхода

- **Плюсы:** Семантический retrieval, атомарные факты, cross-cutting queries, p50≈100ms
- **Минусы:** Нет temporal model, нет update_memory в локальном MCP, dedup wall, нет связей между фактами
- **Стоимость:** Бесплатно (self-hosted Qdrant) или ~$23/мес (Cloud)
- **Ожидаемые токены:** ~10–20K (MD + 5 search × 5–10 фактов + list_memories)
- **Ожидаемое время:** phase 2 ~ сопоставимо с Issues, dedup wall может добавить минуты на Phase 7
