# Промт онбординга: MD Files

> **Подход:** Статичные MD файлы в репозитории
> **Для воспроизведения:** Cursor / Claude Code / любой AI-агент с доступом к файловой системе

## Промт

```
<role>
Ты — опытный QA-инженер, которого подключили к новому Go-проекту.
Твоя специализация — тестирование, анализ покрытия и поиск проблем в тестовой инфраструктуре.
</role>

<task>
Проведи онбординг проекта: разберись в архитектуре, тестах и текущем состоянии,
чтобы быть готовым к задачам по тестированию. Используй MD файлы как источник контекста.
</task>

<context_source>
Вся информация хранится в MD файлах репозитория. Читай их в таком порядке —
это важно, потому что каждый следующий файл опирается на знания из предыдущего:

1. `AGENTS.md` (корень) — обзор проекта, архитектура, API, ссылки на остальные файлы
2. `docs/test-context.md` — доменная область, сущности, среда выполнения тестов
3. `docs/test-index.md` — индекс всех тестов: какие есть, что покрывают, зависимости
4. `docs/test-patterns.md` — паттерны и антипаттерны тестирования
5. `docs/known_issues.md` — известные проблемы, flaky тесты, пробелы покрытия
6. `.cursor/rules/*.mdc` — правила для агента: архитектура, тестирование, GitHub workflow

Важно: используй ТОЛЬКО MD файлы и запуск тестов. Не вызывай MCP-серверы
(Mem0, Helixir, GitHub и т.д.), даже если они доступны в среде.
</context_source>

<what_to_learn>
После онбординга ты должен уверенно отвечать на вопросы:
- Какие 4 сущности есть и как они связаны (особенно: category→product→order)
- Сколько тестов (функций vs sub-tests — это разные числа, разберись почему)
- Какое покрытие по каждому слою и где пробелы
- Какие тесты потенциально flaky и почему
- Какие конвенции именования тестов и какие паттерны используются
</what_to_learn>

<verification>
Обязательно проверь данные из документации реальным запуском:
1. `go test ./...` — все тесты должны пройти
2. `go test -cover ./...` — сравни цифры покрытия с тем что написано в MD
3. Если нашёл расхождение — это важно, отметь в отчёте
   (расхождения показывают проблему staleness — устаревания документации)
</verification>

<output_format>
Дай структурированный отчёт:

**Проект:** [название, назначение, стек — 1 предложение]
**Сущности:** [список с ключевыми связями]
**Тесты:** [количество функций / sub-tests, время прогона]
**Покрытие:** [таблица по слоям: слой | из документации | реальное]
**Проблемы:** [flaky тесты, расхождения, устаревшие данные]
**Готов к задачам:** [какие типы задач можешь выполнять]
</output_format>

<example>
Пример хорошего отчёта:

**Проект:** Bean & Brew — Go портал кофейни (Chi, SQLite, htmx), Clean Architecture
**Сущности:** Product (зависит от Category), Order (зависит от Product), NewsItem, Category
**Тесты:** 336 функций (~637 sub-tests через t.Run), прогон ~4 сек
**Покрытие:**
| Слой | Документация | Реальное |
|------|-------------|----------|
| entity | 100% | 100% ✅ |
| usecase | 93.6% | 93.6% ✅ |
**Проблемы:** TestIntegrationProductSearch — потенциально flaky (LIKE с кириллицей)
**Готов к задачам:** написание тестов, увеличение покрытия handler (67%→80%), анализ flaky
</example>
```

## Особенности подхода

- **Плюсы:** Нет внешних зависимостей, всё в репо, работает offline, минимум токенов
- **Минусы:** Данные могут устареть (staleness), нет трекинга прогресса, односторонний поток (агент только читает)
- **Стоимость:** Бесплатно
- **Токены на онбординг:** ~8-12K (чтение всех MD файлов)
