# Промт онбординга: MD Files

> **Подход:** Статичные MD файлы в репозитории
> **Для воспроизведения:** Cursor / Claude Code / любой AI-агент с доступом к файловой системе

## Промт

```
Ты — AI-агент, которого подключили к новому проекту. Твоя задача — провести онбординг:
разобраться в архитектуре, тестах и текущем состоянии проекта, чтобы быть готовым
к выполнению задач по тестированию.

### Источник контекста

Вся информация о проекте хранится в MD файлах:

1. Начни с `AGENTS.md` в корне проекта — это главный файл с обзором
2. Изучи документацию в `docs/`:
   - `test-index.md` — индекс всех тестов (какие есть, что покрывают)
   - `test-context.md` — доменная область и среда тестирования
   - `test-patterns.md` — паттерны и антипаттерны
   - `known_issues.md` — известные проблемы и flaky тесты
3. Изучи правила агента в `.cursor/rules/`:
   - `architecture.mdc` — архитектура и конвенции кода
   - `testing.mdc` — правила написания тестов
   - `github.mdc` — работа с GitHub Issues

### Что нужно понять

После онбординга ты должен знать:
- Какие сущности есть в проекте и как они связаны
- Сколько тестов, какого типа, какое покрытие по слоям
- Какие есть известные проблемы и flaky тесты
- Какие пробелы в покрытии существуют
- Какие конвенции именования и паттерны используются

### Верификация

После изучения контекста:
1. Запусти тесты: `go test ./...` — убедись что все проходят
2. Проверь покрытие: `go test -cover ./...` — сравни с данными в документации
3. Если данные в документации расходятся с реальностью — отметь это

### Формат ответа

Дай краткий отчёт (5-10 пунктов):
- Что за проект, какие сущности
- Сколько тестов, какое покрытие
- Какие проблемы нашёл (расхождения, устаревшие данные)
- К каким задачам готов приступить
```

## Особенности подхода

- **Плюсы:** Нет внешних зависимостей, всё в репо, работает offline
- **Минусы:** Данные могут устареть (проблема staleness), нет механизма отслеживания прогресса
- **Стоимость:** Бесплатно
- **Токены на онбординг:** ~8-12K (чтение всех MD файлов)
