# Сверка тестового покрытия

Этот skill — точечный инструмент для одной задачи: «по доке coverage X%,
сходится ли с фактом, и если нет — где именно дрейф». Не перепиши тесты,
не объясняй что такое coverage, не предлагай рефакторинг.

## Когда вызывать

- Промт онбординга упомянул Phase «Код-верификация».
- Пользователь спросил «актуальны ли цифры покрытия в MD?» / «правда что
  handler 67%?» / «давай прогон с coverage».
- Готовишь комментарий или правку в тикет, где есть числа покрытия.

## Когда **не** вызывать

- Пользователь просит написать новые тесты — это другая задача (см.
  `.cursor/rules/testing.mdc`).
- Запрос про конкретный flaky / failing тест — это `known_issues.md` и
  диагностика, не сверка цифр.

## Команды

```bash
# Полный прогон, без кеша:
go test ./... -count=1

# Покрытие по пакетам — ОСНОВНАЯ команда сверки:
go test -cover ./...

# Покрытие по функциям внутри пакета (если нужно понять где именно дыра):
go test -coverprofile=/tmp/cover.out ./internal/handler/...
go tool cover -func=/tmp/cover.out
```

Источник истины — вывод `go test -cover ./...`. Числа из MD — снапшот на
дату `Last updated`, не правда.

## Где в проекте лежат заявленные числа

| Файл | Что упоминает |
|------|---------------|
| `AGENTS.md` | сводная таблица coverage по слоям |
| `docs/test-index.md` | сводка в шапке |
| Тело Issue #14 | трекинг-таблица с датой |
| Тело Issue #10 | целевой % handler |
| Тело Issue #13 | целевой % repository |

При расхождении проверь все три верхних — обычно дрейфят синхронно.

## Алгоритм

1. Прочитай заявленные числа из MD/Issue, выпиши себе таблицу `слой → MD%`.
2. `go test -cover ./...`. Сопоставь с таблицей.
3. Для каждого слоя пометь: «совпало» / «MD устарел на N п.п.» / «MD выше факта на N п.п.».
4. Если расхождение ≥ 2 п.п. — это дрейф, требует обновления MD. Меньше — округление, не трогай.
5. Если в каком-то слое 0% при ненулевом ожидании — значит файл/пакет вообще не тестируется (см. `pages.go`, `auth.go` для handler сейчас). Это не «обновить число», это материал для тикета (см. #10).

## Что обновлять при подтверждённом дрейфе

В одном проходе, не россыпью:

- `AGENTS.md` — таблица coverage и `Last updated` на сегодня.
- `docs/test-index.md` — строка coverage в «Сводке» и `Last updated`.
- Тело трекинг-issue (#14) через `gh issue edit 14 --body ...`.
- В `docs/known_issues.md` — **только** если дрейф устойчивый (повторился
  между прогонами) или вскрыл новый пробел покрытия. Без номеров тикетов
  (см. правило в `known_issues.md`).

Что **не** делать:
- Не править coverage-числа в теле тикетов с label `context:*` без
  явного запроса — это living docs.
- Не закрывать тикеты по факту «coverage совпал» — закрытие через
  одобрение пользователя (см. `.cursor/rules/testing.mdc`).
- Не комментировать в тикет «прогнал, всё ок» — повторная верификация
  не повод для шума (см. `.cursor/rules/github.mdc`).

## Формат отчёта о сверке

Короткая таблица для человека, без воды:

```
Сверка coverage на YYYY-MM-DD:

| Слой              | По MD  | Факт  | Расхождение     |
|-------------------|--------|-------|-----------------|
| entity            | 100%   | 100%  | совпало         |
| usecase           | 93.6%  | 93.6% | совпало         |
| repository/sqlite | 77.5%  | 77.5% | совпало         |
| handler           | 67%    | 67.0% | совпало         |
| cmd/server        | 0%     | 0%    | совпало (main)  |
```

Если всё совпало — одной строкой «MD соответствует фактическому покрытию,
правок не нужно». Без перечисления каждой строки и без «прогнал X тестов
за Y секунд» — это шум.
