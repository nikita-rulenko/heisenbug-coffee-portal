# Reference: подход GitHub Issues

Дельта подхода GitHub Issues к skill-у `qa-onboarding`. Применяется
когда пользователь запустил промт `prompts/show_difference_v2/git_iss_onb.md`.

> Имена `gh`-команд, таксономия меток репо, правила комментариев и
> закрытия, формат чекбоксов — в `.cursor/rules/github.mdc`. Этот
> reference — только про то, **что обязан показать отчёт**, чтобы
> подход был видим и отличим от чистого MD.

## Уникальная сила подхода — и за что её засчитают

Issues — **workflow-слой** поверх MD: номера, цитаты из тел тикетов,
labels, чекбоксы прогресса, авторы, open/close timestamps, timeline,
`stateReason`, связи с PR. Всё это в MD физически отсутствует.

В этом подходе агент обогащает MD-фундамент данными из тикетов:
кто что делает, в каком статусе, какой приоритет, что закрыто, а
что закрыто-но-не-готово.

Промт может прямо просить искать workflow-сигнал в тикетах. Но он
**не должен** запрещать пользоваться MD и реальностью — фундамент
фактов всё равно из MD.

**Главный кейс подхода — `closed ≠ done`:** тикет закрыт, чекбокс
стоит, метка «completed» — а код говорит обратное. Это единственная
находка, которую MD принципиально не может сделать, потому что у
MD нет workflow-состояния.

Если отчёт не цитирует тикеты дословно, не различает `closed=done`
и `closed≠done`, не переводит labels прозой и не ссылается ни на
один timeline-сигнал — значит агент использовал Issues как
дополнительный README и методология проиграла MD (дороже, медленнее,
тот же результат).

Репозиторий: `nikita-rulenko/heisenbug-coffee-portal`.

## Источники

1. **MD (фундамент):** `AGENTS.md`, `docs/project_overview.md`,
   `docs/test-context.md`, `docs/test-index.md`,
   `docs/test-patterns.md`, `docs/known_issues.md`,
   `.cursor/rules/architecture.mdc`, `.cursor/rules/testing.mdc`,
   `.cursor/rules/github.mdc`.
2. **GitHub Issues:** через MCP (`list_issues`, `issue_read:get`,
   `issue_read:get_comments`, `issue_read:get_labels`) или `gh` CLI
   (см. R2 ниже). MCP не отдаёт timeline-события — для R6 используй
   `gh api .../issues/N/timeline`.
3. **PR / коммиты:** `gh pr view`, `git log --all --source -- <file>`
   — для детекции revert-а после `closedAt`.
4. **Реальность:** `go test -cover ./...`, `grep`, `Read`, `ls`.

## Обязательные артефакты в отчёте

Восемь требований. Каждое — это то, без чего онбординг «через
Issues» не засчитывается.

### R1. Каждое упоминание тикета — кликабельная ссылка с цитатой

В прозе и в списке задач каждый тикет упоминается **только** как
кликабельная ссылка с номером и короткой цитатой из тела/AC/чекбокса:

```
[#10 "Coverage handler ≥ 80%"](https://github.com/nikita-rulenko/heisenbug-coffee-portal/issues/10)
```

Голый `#10` без ссылки и цитаты — запрещён. Цитату бери **дословно**
из `body` или из `- [x]`-чекбокса, в кавычках. Если цитата не
помещается в 8 слов — обрежь многоточием.

В буллете-находке рядом с тикетом всегда указан state и `stateReason`:
«закрыт как `completed` 2026-04-12, PR #42» — это и есть workflow-
данные, которых нет в MD.

В разделе «Готов к работе. Список задач» каждый пункт, привязанный
к тикету, — тоже кликабельная ссылка:

```
- **P1** — довести handler-покрытие до 80% по
  [#10 "Coverage handler ≥ 80%"](https://github.com/nikita-rulenko/heisenbug-coffee-portal/issues/10):
  сейчас 67% (verified: `go test -cover ./internal/handler`),
  фикс откатили. Начать с `internal/handler/pages.go` (0%).
```

Не `**#10** (open)` и не `— #10 — …`. Инженер читает «Готов к
работе» и должен одним кликом открыть тикет.

### R2. Минимум 5 `gh` / MCP вызовов в probe-фазе

Проверяемый набор (пропустишь — подход использован вполсилы):

1. **Общий снимок:** `gh issue list --state all --limit 50 --json number,title,state,stateReason,labels,closedAt,updatedAt,closedByPullRequestsReferences` или MCP-эквивалент `list_issues`.
2. **Тело+комментарии минимум 2 закрытых тикетов:** `gh issue view N --json body,comments,labels,state,stateReason,closedAt,closedByPullRequestsReferences` — это база для closed≠done верификации.
3. **Timeline хотя бы одного «спорного» тикета:** `gh api repos/nikita-rulenko/heisenbug-coffee-portal/issues/N/timeline` — labeled/unlabeled/closed/reopened/cross-referenced события. MCP `issue_read` timeline не возвращает, поэтому здесь только `gh api`.
4. **Метки целиком:** `gh label list --json name,description,color` или `list_labels` — основа для перевода labels прозой (R5).
5. **PR, закрывшие тикеты:** проверь поле `closedByPullRequestsReferences` из пункта 1; если нужны детали — `gh pr view <num> --json number,title,mergedAt,mergeCommit,files`.

### R3. Каждый закрытый тикет в отчёте получает verdict «closed=done / closed≠done»

Для каждого тикета, на который ссылаешься в прозе, приведи
**явный вывод**:

- `closed=done`: AC из тела / чекбоксы фактически выполнены, код /
  `go test` подтверждают. Достаточно одной строки prose.
- `closed≠done`: чекбокс стоит или AC сформулирован, но реальность
  расходится. В буллете явно указано **что именно** не сошлось +
  инлайн-команда проверки + причина из таксономии.

Если закрытый тикет в отчёте не получил ни одного из двух verdict —
значит Issues использован как README, и вся цель подхода упущена.

### R4. Минимум 3 чекбокса процитированы дословно

Чекбоксы `- [x]` / `- [ ]` — главный машинно-читаемый сигнал
прогресса в теле issue. Не процитировать их = не использовать
Issues.

Формат: цитата чекбокса из тикета `[#N "..."](url)` → результат
проверки `(verified: ...)` → verdict + причина.

Если в репо не 3 чекбокса в релевантных тикетах — отметь это
явно («в открытых релевантных тикетах чекбоксов нашлось только N,
больше цитировать нечего»), но **не молчи**.

### R5. Каждая label переводится в речи при первом упоминании

Сырой label-синтаксис запрещён как «материал отчёта». При первом
упоминании метки в прозе — перевод + (опционально) оригинал в
скобках один раз:

- `context:coverage` → «метка про coverage-цели»
- `area:handler` → «область handler-слоя»
- `kind:bug` → «тип: баг»
- `status:blocked` → «статус: заблокировано»

Дальше по тексту — только человеческий вариант.

### R6. Раздел «Open → closed timeline»

Мини-секция 3–6 буллетов про временную ось релевантных тикетов.
Данные — из `createdAt` / `closedAt` / `updatedAt` + timeline events
(`labeled`, `closed`, `reopened`, `cross-referenced`):

- `[#10 "Coverage handler ≥ 80%"](url)` открыт 2026-03-28, закрыт
  2026-04-12 (`stateReason=completed`, PR #42). После закрытия —
  коммит-revert 2026-04-13 (verified: `git log --oneline -- internal/handler`).
  Сигнал `rolled_back_fix`.
- `[#8 "..."](url)` открыт 2026-04-03, последнее обновление
  2026-04-05, висит open. Коммит с тестом по AC — 2026-04-10.
  Сигнал `partial_update` (тикет забыли закрыть).

Это тот signal, которого в MD физически нет: человеческая
временная ось (кто закрыл, когда, чем — PR или руками).

### R7. Каждый «closed» verdict ссылается на `stateReason` и `closedByPullRequestsReferences`

Две эти штуки — единственный machine-readable ответ на вопрос «как
именно закрыт тикет». Красные флаги:

- `stateReason=completed`, но `closedByPullRequestsReferences=[]`
  — закрыли руками без PR. Подозрительно: либо тикет-дубль, либо
  доработка вне VCS.
- `stateReason=not_planned` — AC больше не актуален, сверять с
  кодом бессмысленно, но это надо заметить, а не верифицировать
  как «done».
- Повтор `closed` → `reopened` → `closed` за короткий период —
  кандидат на `transient_state`.

### R8. Минимум 2 буллета опираются на комментарии, не на тело

Поток комментариев — место, где протекает реальность: «fixed in
abc1234», «reverted», «blocked by #N», «QA said still fails».
Отчёт без единой цитаты из комментариев = Issues использован как
статическое хранилище.

В сводке можно (необязательно) расщеплять: `Issues-body:b1 +
Issues-comments:b2` внутри общего `Issues:b`, если для тебя это
принципиально важно. Минимум — 2 буллета, где источник — именно
комментарий (включая имя автора и дату).

## Главный раздел отчёта: «Что Issues добавили поверх MD»

Идёт сразу после «Что я понял при онбординге» (или после
опциональной таблицы покрытия). Формат — **3–5 буллетов**. Каждый
буллет — конкретный workflow-факт из тикета (статус, чекбокс,
комментарий, PR-связь, автор), которого в MD нет. В каждом буллете
— кликабельная ссылка по R1, цитата по R3/R4, инлайн-команда
проверки, и (если расхождение) — причина из таксономии.

Жёсткое правило цитирования: **каждый буллет содержит саму
находку** (цитату чекбокса, цитату комментария, конкретный
timeline-сигнал), а не мета-описание («в тикетах есть workflow-
данные»).

После списка — одна строка-итог: чему верить (код / тикеты / MD),
какие тикеты в первую очередь нужно переоткрыть/закрыть.

## Контрастная пара примеров

**ПЛОХО** (упущена цитата, URL, `stateReason`, причина):

> - Тикет #10 закрыт, но coverage handler не дотянут до 80%.

Голый `#10`, нет ссылки, нет цитаты чекбокса, нет PR, нет verdict.

**ХОРОШО** (вся Issues-специфика на месте):

> - [#10 "Coverage handler ≥ 80%"](https://github.com/nikita-rulenko/heisenbug-coffee-portal/issues/10)
>   закрыт как `completed` 2026-04-12 после PR #42, чекбокс
>   `- [x] Coverage handler ≥ 80%` проставлен — но
>   `go test -cover ./internal/handler` даёт 67.2%, `pages.go` 0%
>   (verified: `go test -cover ./internal/handler -coverpkg=./internal/handler/...`).
>   В комментарии автора PR — «fixed in a1b2c3», в `git log` —
>   revert через сутки после закрытия. Классический
>   `rolled_back_fix`: работу приняли, метрики зафиксировали, код
>   откатили, тикет никто не переоткрыл. **Verdict: closed≠done.**

## Подстановки для сводки

- `<источники>` в первой строке сводки: `MD + Issues`.
- Арифметика: `MD:a + Issues:b + probe:p = Проверено`.
- Опционально внутри `Issues:b`: `body:b1 + comments:b2`, если
  хочешь подчеркнуть, что комментарии реально использовались (R8).
- Заголовок раздела с находками: «Что Issues добавили поверх MD».

## Сигналы причинной таксономии (шаг Classify)

Берёшь из SKILL.md 5 категорий + `source_behind_reality`. Issues-
специфичные сигналы:

- **`rolled_back_fix`:** closed `completed` → коммит-revert после
  `closedAt`; комментарий «fixed in abc» + revert в `git log`; в
  связанных PR есть `Revert "…"` заголовок.
- **`doc_drift`:** тикет или MD отстали от кода, следов revert нет,
  `closedAt` старше последнего коммита по файлу. Или `closed` +
  `stateReason=not_planned` (AC просто устарел).
- **`partial_update`:** часть синхронизирована, часть — нет.
  Классика: тесты написаны, но тикет open; или три из пяти
  чекбоксов проставлены, остальные нет, а код говорит что всё
  сделано.
- **`transient_state`:** open↔closed скачки, свежий `reopened` в
  timeline, активные коммиты туда-сюда в окрестности даты.
- **`unknown`:** данных нет — ставь, не додумывай.
- **`source_behind_reality` (probe-находка):** реальность шире
  тикетов. Пример: в `internal/` четыре слоя, а `context:coverage`-
  тикеты заведены только на один.

## Writes-политика

Онбординг — репорт. Во время сессии writes не производятся.

**Запрещено:**

- `update_issue` — в том числе для отметки чекбоксов, даже если по
  коду «сделано».
- `close_issue` / `reopen_issue`.
- `add_comment` / `add_issue_comment` — никаких «прогнал тесты,
  вот результат» в тикетах.
- Любые Edit/Write по MD/docs/коду/конфигам.

Причина изоляции: если онбординг чинит тикеты на лету, следующий
подход увидит уже прибранный workflow и пропустит расхождение —
бенчмарк ломается.

Если нашёл ошибку в `gh` / MCP / тикетах, помешавшую онбордингу —
одной строкой в «Найденные неточности и ограничения Issues».

## Изоляция MCP

Не вызывай `mem0-local`, `user-helixir-local` / `helixir-local` —
даже если подключены. Подход Issues должен оставаться чистым:
иначе нельзя сказать, чьё открытие — тикетов или графа/памяти.

## Self-check перед финишем

1. Каждое упоминание тикета — кликабельная ссылка `[#N "цитата"](url)`?
   Голые `#10` без ссылки = правь (R1).
2. В разделе «Готов к работе. Список задач» **каждый** пункт с
   привязкой к тикету — кликабельная ссылка по тому же шаблону (R1).
3. Каждый упомянутый в прозе закрытый тикет получил verdict
   `closed=done` или `closed≠done` (R3)?
4. Процитировано ≥3 чекбокса дословно (R4)?
5. Все labels переведены на русский при первом упоминании (R5)?
6. Есть раздел «Open → closed timeline» минимум на 3 тикета (R6)?
7. Хотя бы 2 буллета из комментариев, а не из тел (R8)?
8. Каждое число в прозе — с инлайн-командой проверки `(verified: …)`?
9. Никаких трейс-таблиц в отчёте.
10. Маленькая таблица покрытия — только если есть информативная
    разница хотя бы по одной строке.
11. «Готов к работе» — каждый пункт **P1**/**P2**/**P3** с
    действием и «где начинать»; минимум 1 × P1 при расхождении ≥ 1.

Любой «нет» = правь раздел, а не сводку. Сводка подгоняется под
наполнение разделов, не наоборот.
